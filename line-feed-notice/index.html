PDS4 Line Feed Changes

The Planetary Data System is considering a change to its standards, which may impact users of PDS data. If and how PDS proceeds with this change depends in large part on the response of the community. See below for contact information.

What would change?

Current standards require that Text Stream (like ASCII), Character Table (like Fixed-Width Tables), and Delimited Table (like CSV) files follow the Windows format in which every line ends with both the Carriage Return (CR) and Line Feed (LF) characters. The change under consideration would allow such archived data in which every line ends with a Line Feed character (LF) alone, used by Linux and MacOS systems, in addition to the prior Windows format.

What are line feed characters?

In digital files, special invisible "control characters" are used to indicate when a line has ended. These may be used in a text document, table, image, or any other format. In Windows and most older computing systems, this is indicated with two adjacent characters: "Carriage Return" (CR, Hex 0xD, "\r") and "Line Feed" (LF, Hex 0xA, "\n"). On Unix-based systems like Mac and Linux, this is typically represented as a single Line Feed character instead. Further background can be found here.

How would this affect me?

If you use your own custom software or scripts to parse and use data from the PDS, this may need to be updated to support this change.

If you use software distributed by the PDS on our archived data, you would need to download the latest versions when those are available at the same time this change is made (date TBD).

What might I need to change in my code?

Maybe nothing! If you use widely-distributed image/text/file parsing libraries in your code, many of them already know how to handle variable format line endings.

Otherwise, you would need to inspect your code for any place you are expecting a CR+LF newline, and modify it to expect either CR+LF or LF. Note that errors might occur if you switch to just expecting LF; the trailing CR could cause issues. You must handle both cases.

In many programming languages (like Python), these characters are represented in code by "\n" (for LF) and "\r" for carriage return. So in those cases, you would need to change any code that expected "\r\n" to also expect "\n".

What do I not need to worry about?

Data that has already been archived with the existing PDS3 or PDS4 standards would not change, and can continue to be used with any tools out there today. This change would only affect some data files archived in the future model versions (date TBD).

Software written and distributed by the PDS would be updated to work with this new format at the same time, so you would not need to modify that software, merely use the most updated versions.

Data providers already supplying data using Windows-format line endings will not need to change anything. The record delimiter type has always been specified in the accompanying labels, and will continue to be so.

I am a data provider, how would this affect me?

This is merely an addition to what's allowed in our standards. Any existing data that you provide would continue to be archivable. Going forward though, you would have more flexibility in the types of files you can archive, or the amount of work you need to put in to make your data archive-ready.

Who can I contact for feedback or questions?

Please contact us at sbn@psi.edu

We are seeking community feedback on whether or not we should move forward with this change. If this change would impact you, we would like to know if you support or oppose this change. We are also here if you have further questions, or need help understanding how this would impact any tools you use or have written.
